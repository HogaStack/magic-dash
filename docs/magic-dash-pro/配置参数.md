# `magic-dash-pro`配置参数

`magic-dash-pro`的配置集中在`configs/`目录。它继承多页面、持续扩展模板的配置方式，并额外增加面向复杂用户与部门关系的鉴权、会话、安全、邮件、`OTP`和数据库配置。`Flask`与`FastAPI`后端变体使用相同的邮件验证码、`OTP`动态口令登录配置，以及配置参数与数据库规则表混合的权限管理机制。

## `configs/email_config.py`

邮件验证码登录默认关闭。启用前需要将`BaseConfig.enable_email_login`设置为`True`，并配置可用的`SMTP`服务：

| 参数 | 类型 | 默认值 | 说明 |
| --- | --- | --- | --- |
| `smtp_server` | `str` | `""` | `SMTP`服务地址，例如`smtp.163.com`或`smtp.qq.com` |
| `smtp_port` | `Optional[int]` | `None` | `SMTP`服务端口，必须是`1`至`65535`之间的整数 |
| `sender_email` | `str` | `""` | 用于发送验证码的邮箱地址，也是`SMTP`认证用户名 |
| `sender_password` | `str` | `""` | 发件邮箱的`SMTP`授权码或服务商要求的凭据，通常不是邮箱登录密码 |
| `sender_name` | `str` | `BaseConfig.app_title` | 验证码邮件中展示的发件人名称 |
| `smtp_use_ssl` | `bool` | `False` | 是否在建立连接时直接使用`SSL`，通常对应465端口 |
| `smtp_use_starttls` | `bool` | `False` | 是否先建立普通连接再升级为`STARTTLS`，通常对应587端口 |
| `smtp_timeout` | `int` | `10` | `SMTP`连接超时时间，单位为秒，必须大于0 |
| `verification_code_expire_seconds` | `int` | `300` | 验证码有效期，单位为秒，必须为正整数 |
| `verification_code_resend_interval_seconds` | `int` | `60` | 同一邮箱重复发送验证码的等待时间，单位为秒，必须为正整数且不能超过验证码有效期 |

`smtp_use_ssl`和`smtp_use_starttls`不能同时开启。常见连接方式如下：

| 连接方式 | 常见端口 | `smtp_use_ssl` | `smtp_use_starttls` |
| --- | --- | --- | --- |
| `SSL` | `465` | `True` | `False` |
| `STARTTLS` | `587` | `False` | `True` |
| 普通连接 | `25` | `False` | `False` |

例如，使用`SSL`连接时可配置为：

```python
class EmailConfig:
    smtp_server = "smtp.example.com"
    smtp_port = 465
    sender_email = "no-reply@example.com"
    sender_password = "请替换为SMTP授权码"
    sender_name = BaseConfig.app_title
    smtp_use_ssl = True
    smtp_use_starttls = False
    smtp_timeout = 10
    verification_code_expire_seconds = 300
    verification_code_resend_interval_seconds = 60
```

模板会使用上述账号登录`SMTP`服务，并发送同时包含纯文本和`HTML`内容的验证码邮件。邮件主题、正文中的应用名称和发件人默认名称均来自`BaseConfig.app_title`。

## `configs/otp_config.py`

`OTP`动态口令登录默认关闭。启用前需要将`BaseConfig.enable_otp_login`设置为`True`，并确保已安装模板依赖中的`pyotp`和`cryptography`：

| 参数 | 类型 | 默认值 | 说明 |
| --- | --- | --- | --- |
| `issuer_name` | `str` | `BaseConfig.app_title` | 认证器`App`中展示的发行方名称 |
| `otp_digits` | `int` | `6` | 动态口令位数，模板固定要求为6位数字 |
| `otp_interval_seconds` | `int` | `30` | `TOTP`动态口令刷新周期，单位为秒，必须为正整数 |
| `otp_valid_window` | `int` | `1` | 校验时允许前后偏移的时间窗口数量，用于兼容客户端与服务器的轻微时钟偏差 |
| `max_failed_attempts` | `int` | `5` | 连续校验失败达到该次数后，临时锁定该用户的`OTP`登录 |
| `lockout_seconds` | `int` | `300` | `OTP`登录临时锁定时长，单位为秒，必须为正整数 |
| `secret_length` | `int` | `32` | 生成`OTP`共享密钥时使用的`Base32`字符串长度，不能小于32 |
| `secret_crypto_key` | `str` | `""` | `OTP`共享密钥落库加密材料；为空时回退使用`BaseConfig.app_secret_key`，生产环境建议设置为独立的高强度随机字符串 |

启用后，用户需要先登录系统，在右上角用户菜单进入“OTP绑定”，使用认证器`App`扫码并输入当前6位动态口令完成绑定。绑定成功后，登录页“更多登录方式-OTP动态口令登录”将使用用户名和认证器`App`中的6位动态口令建立普通登录会话。

`OTP`凭据保存在`models/otp_credentials.py`对应的数据表中，共享密钥会加密落库。连续错误、临时锁定、同一时间窗口口令防重放和登录结果记录由模板内置逻辑处理。

## `configs/auth_config.py`

`AuthConfig`定义权限系统的硬编码基线。运行时页面鉴权、用户管理角色选项、侧边菜单和页面搜索不会直接只读取该配置，而是通过`UserPermissionGroups`把硬编码配置与数据库权限组综合后再使用。

| 参数 | 类型 | 默认值 | 说明 |
| --- | --- | --- | --- |
| `roles` | `dict` | `admin`、`normal` | 系统角色定义 |
| `normal_role` | `str` | `"normal"` | 常规用户角色标识 |
| `admin_role` | `str` | `"admin"` | 管理员角色标识 |
| `pathname_access_rules` | `dict` | `admin`全部可访问，`normal`排除登录日志 | 不同角色对页面菜单`key`的访问规则 |

`pathname_access_rules`支持三种规则：

| 类型 | 说明 |
| --- | --- |
| `all` | 可访问全部页面 |
| `include` | 仅可访问`keys`中声明的页面；首页会自动纳入 |
| `exclude` | 可访问除`keys`外的页面 |

### 配置参数与数据库混合权限

模板会把`AuthConfig.roles`、`AuthConfig.pathname_access_rules`和`models/user_permission_groups.py`对应的数据库权限分组表合成为最终生效的角色与页面访问规则。`Flask`和`FastAPI`后端在上层行为上保持一致，根路由都会通过`UserPermissionGroups.get_effective_pathname_access_rule(current_user.user_role)`校验当前用户访问目标页面的权限。

关键规则如下：

- `AuthConfig.admin_role`对应的管理员角色始终以硬编码配置为准，不能通过数据库权限组覆盖、编辑或删除；运行时也会清理历史遗留的同`id`数据库记录。因此管理员角色必须同时存在于`AuthConfig.roles`和`AuthConfig.pathname_access_rules`中，并在硬编码规则里明确授予预期的管理员权限。
- `AuthConfig.roles`中的非管理员角色默认使用硬编码名称与`pathname_access_rules`；如果数据库中存在同`permission_group_id`的有效记录，则该记录会覆盖该角色的展示名称和页面访问规则。删除这条数据库记录后，会恢复为硬编码配置。
- 数据库新增权限组可作为新的用户角色使用，前提是权限组`id`和名称不与硬编码角色冲突，且页面访问规则合法。
- 数据库权限组的`access_rule_keys`只能引用`RouterConfig.valid_pathnames`中已登记、且不属于`RouterConfig.public_pathnames`的字符串页面路径。正则通配页面规则无法通过当前权限管理界面写入数据库，如需控制通配页面，建议继续在`AuthConfig.pathname_access_rules`中硬编码维护。
- 访问规则为`include`时，首页`/`和`RouterConfig.index_pathname`不受限制，仍会自动放行；公开页面会在登录和权限校验前放行。
- 当前用户角色不属于配置参数或数据库合成后的有效角色时，受保护页面访问会被重定向到`403`页面。

数据库权限组表字段如下：

| 字段 | 说明 |
| --- | --- |
| `permission_group_id` | 权限组主键，对应`Users.user_role` |
| `permission_group_name` | 权限组展示名称，要求唯一 |
| `access_rule_type` | 页面访问规则类型：`all`、`include`或`exclude` |
| `access_rule_keys` | 页面访问规则`key`列表，数据库记录中为字符串路径列表 |
| `is_builtin` | 是否为硬编码角色对应的数据库覆盖记录 |
| `other_info` | 预留扩展信息，允许保存任意`JSON`数据 |

管理员登录后，可在右上角用户菜单进入“权限管理”，查看硬编码权限组、数据库权限组、冲突状态和实际生效规则。该页面支持新增数据库权限组、编辑非管理员权限组、删除数据库权限组或删除硬编码角色的数据库覆盖记录。普通用户的可选角色、用户列表中的角色名称、页面搜索和侧边菜单都会使用合成后的有效角色与权限规则。

常见冲突状态含义如下：

| 状态 | 含义 | 生效结果 |
| --- | --- | --- |
| 正常 | 权限组未与硬编码配置冲突，且规则合法 | 正常纳入有效角色 |
| 数据库覆盖 | 数据库中存在同`id`的非管理员硬编码角色记录 | 使用数据库中的名称和访问规则覆盖硬编码基线 |
| 名称冲突 | 数据库权限组名称与其他硬编码角色名称冲突 | 冲突数据库记录不纳入有效角色；同`id`覆盖记录会被忽略 |
| 规则异常 | 数据库权限组访问规则类型或`keys`不合法 | 冲突数据库记录不纳入有效角色 |
| `admin`数据库记录禁用 | 数据库中存在`AuthConfig.admin_role`同`id`记录 | 该记录会被清理，管理员权限始终以硬编码配置为准 |

## 配置修改原则

- 新增受保护核心页面时，通常需要同步维护`RouterConfig.core_side_menu`、`RouterConfig.valid_pathnames`、`components/page_content.py`中的页面渲染分发逻辑，以及`AuthConfig.pathname_access_rules`或数据库权限组中的访问规则。
- 若希望该页面可在“权限管理”中被数据库权限组选择，页面`pathname`必须以字符串形式登记到`RouterConfig.valid_pathnames`，且不能加入`RouterConfig.public_pathnames`。
- 新增公开页面时，需要维护`RouterConfig.valid_pathnames`和`RouterConfig.public_pathnames`，并在`app.py`的根路由中补充该公开路径对应的渲染分支。
- 新增独立页面时，需要将`pathname`或正则对象加入`RouterConfig.independent_core_pathnames`，并在`views/core_pages/__init__.py`中补充独立渲染分支。
- 新增通配页面时，先在`RouterConfig.wildcard_patterns`中定义正则，再把对应正则对象加入`RouterConfig.valid_pathnames`；若需要嵌入核心布局，还要同步调整核心页面路由回调和`components/page_content.py`。
- 调整侧边菜单层级或权限规则后，检查`RouterConfig.side_menu_open_keys`、`AuthConfig.pathname_access_rules`和数据库权限组记录，确保菜单展开与角色访问范围符合预期。

## 安全建议

- 生产环境务必替换`app_secret_key`。
- 不要将真实的`sender_password`提交到版本控制；生产环境建议通过环境变量或密钥管理服务注入该值。
- 启用`OTP`动态口令登录时，生产环境建议显式设置独立的`OtpConfig.secret_crypto_key`，并妥善备份；密钥变更会影响已绑定`OTP`凭据的解密。
- 优先使用`SSL`或`STARTTLS`加密`SMTP`连接，并以邮件服务商提供的地址、端口及授权方式为准。
- `verification_code_expire_seconds`控制验证码暴露窗口，`verification_code_resend_interval_seconds`单独控制用户再次获取验证码前的等待时间。
- 模板不记录或限制单个验证码的错误校验次数；面向公网部署时，建议在反向代理或网关层对验证码校验请求补充按`IP`和邮箱维度的限流与监控。
- 多套应用部署在同一域名或主机地址下时，为每套应用设置不同的`app_session_cookie_name`和`session_token_cookie_name`。
- 开启`enable_login_rsa_crypto`时**必须**配合`HTTPS`环境使用；非`HTTPS`环境不应启用该配置。
- 初始化后的默认管理员密码应尽快修改。
- 谨慎授予“权限管理”入口。模板默认仅`AuthConfig.admin_role`用户可见该入口，且`admin`权限组不能被数据库记录覆盖，以降低误改全局权限的风险。
- 不要移除或弱化硬编码管理员规则。管理员权限必须在`AuthConfig.roles`和`AuthConfig.pathname_access_rules`中明确体现，数据库权限组不能为`admin`补充或修复缺失的权限规则。
